# Pico + W65C816S bus firmware

Raspberry Pi Pico/RP2040 が外付け W65C816S に ROM、RAM、USB CDC UART を提供するための Pico SDK ファームウェアです。GPIO29 からクロックを供給し、GPIO26/27 は固定の非アクティブ制御信号として出力します。

## Pin map

| Pico GPIO | Direction | Signal |
| --- | --- | --- |
| 0-15 | in | Address bus A0-A15 |
| 16-23 | in/out, PIO controlled | Data bus D0-D7 |
| 24 | out | /NMI |
| 25 | in | R/W |
| 26 | out | /IRQ, fixed high |
| 27 | out | RDY, fixed high |
| 28 | out | /RESET |
| 29 | out, PIO controlled | MPU clock |

Standard Pico boards do not bring every GPIO in this table out as a normal header pin. This pinout is most natural on a bare RP2040 or a board that exposes GPIO24 and GPIO29.

Tie any W65C816S control inputs that are not driven by this firmware, such as BE and /ABORT, to their inactive external levels as required by your board.

## Memory map

| MPU address | Function |
| --- | --- |
| `$0000-$1fff` | 8KB RAM, initialized to `$00` |
| `$4018` | 6850-like status register |
| `$4019` | 6850-like data register |
| `$6000-$7fff` | 8KB RAM, initialized to `$00` |
| `$8000-$ffff` | 32KB ROM image |
| other addresses | unmapped, reads return `$ff`, writes ignored |

The MPU-visible memory image is one 64KB SRAM-resident array. It is initialized to `$ff`, then the two independent RAM windows at `$0000-$1fff` and `$6000-$7fff` are initialized to `$00`. The ROM image is inserted into the upper half of the array:

```c
static uint8_t memory[64 * 1024] __attribute__((section(".data.memory_image"), aligned(4), used)) = {
    [0 ... MEMORY_SIZE - 1] = 0xff,
    [0 ... RAM_SIZE - 1] = 0x00,
    [MPU_RAM2_BASE ... MPU_RAM2_BASE + RAM_SIZE - 1] = 0x00,
    [0x8000] =
#include "rom_image.h"
};
```

`src/rom_image.h` currently includes `../../ROM/ROM6502.h`. Replace it with up to 32KB of comma-separated hex bytes if you want a private 65816 Pico ROM image. Offset `$0000` in that file maps to MPU address `$8000`.

## UART status bits

The `$4018` read status register only exposes the two MC6850-compatible flow bits needed by the bus loop:

| Bit | Meaning in this firmware |
| --- | --- |
| 0 | RDRF: USB-to-MPU SIO FIFO has at least one byte |
| 1 | TDRE: the one-command core1-to-core0 slot is empty |

RDRF maps to the RP2040 SIO receive FIFO `VLD` bit. TDRE is set while the shared one-command core1-to-core0 slot is empty. Writes to `$4018` are ignored. Interrupt output, DCD/CTS, and overrun status are not implemented.

## Timing model

Core 1 is dedicated to the MPU bus loop. The PIO clock state machine pushes two complete GPIO snapshots into its RX FIFO during every MPU cycle: one immediately after the clock rises and one immediately before it falls. The bus loop uses the first snapshot for address/R/W decoding and the second for write data. This removes timing-dependent SIO polling and avoids sampling the W65C816S data bus after it has returned to the low-phase bank-address state. Core 1 acknowledges each snapshot through the clock state machine TX FIFO. At the end of each high or low phase, the state machine performs a blocking pull; an ACK that arrived during the normal processing window passes immediately, while a late ACK extends the current phase.

The data GPIOs are assigned to PIO, not SIO. The data bus uses three PIO state machines: one continuously accepts low-byte values and writes GPIO16-23, one controls GPIO16-19 directions, and one controls GPIO20-23 directions. The two 4-bit direction SMs watch GPIO29 and R/W directly; when the clock is high and R/W is high they drive their nibbles, then wait 10 PIO cycles after the clock falls before releasing GPIO16-23 to input/Hi-Z. At 232MHz that post-fall hold is about 43.1ns, keeping it close to the W65C816S 40ns read-data hold requirement.

Core 0 owns USB CDC stdio. USB-to-MPU input continues to use the RP2040 SIO FIFO, whose `VLD` bit is ACIA RDRF. Core1-to-core0 ACIA output and PPU commands use a shared one-command slot: the first command queues immediately, while a second command blocks until core 0 finishes processing the previous command and clears the slot. The empty slot is reported as ACIA TDRE. MPU reads from `$4019` never block; MC6850 software is expected to check RDRF first, and a read while empty intentionally returns an undefined value.
VBlank timing is counted on core 1 and delivered to core 0 as separate begin and end commands. Core 0 owns the PPU VBlank status bit and drives NMI low whenever both the VBlank status bit and the PPUCTRL NMI-enable bit are set.

GPIO29 clock output is generated by PIO instead of `clock_gpio_init()`, because RP2040 hardware clock outputs are limited to selected GPOUT pins. The firmware sets VREG to 1.15V and `clk_sys` to 232MHz. The clock SM uses `clkdiv = 1 + 232/256` and 34 PIO cycles per half-period, producing `232MHz / (1 + 232/256) / (34 + 34)`, about 1.78977MHz.

## Build

Install and export the Pico SDK in the usual way, then build:

```sh
export PICO_SDK_PATH=/path/to/pico-sdk
cmake -S . -B build -DPICO_BOARD=pico
cmake --build build
```

The UF2 will be generated as:

```text
build/pico_mpu_65816.uf2
```

For timing validation, build with optimization enabled and check GPIO29 clock, R/W, GPIO16-23, /RESET, /NMI, /IRQ, and RDY on a logic analyzer. The final margin depends on the exact board, wiring, level shifting, and W65C816S timing.
