# Pico + MPU 6802 bus firmware

Raspberry Pi Pico/RP2040 が外付け MPU 6802 に ROM、RAM、USB CDC UART を提供するための Pico SDK ファームウェアです。

## Pin map

| Pico GPIO | Direction | Signal |
| --- | --- | --- |
| 0-15 | in | Address bus |
| 16-23 | in/out, PIO controlled | Data bus |
| 24 | in | E, 2MHz returned clock |
| 25 | in | R/W |
| 26 | in | VMA |
| 27 | unused | N.C. |
| 28 | out | /RESET |
| 29 | out, PIO controlled | 8MHz clock |

Standard Pico boards do not bring every GPIO in this table out as a normal header pin. This pinout is most natural on a bare RP2040 or a board that exposes GPIO24 and GPIO29.

## Memory map

| MPU address | Function |
| --- | --- |
| `$0000-$7fff` | 32KB RAM, zero-filled |
| `$8000` | 6850-like status register, writes ignored |
| `$8001` | 6850-like data register |
| `$8002-$bfff` | unmapped, reads return `$ff`, writes ignored |
| `$c000-$ffff` | 16KB ROM |

The MPU-visible memory image is one 64KB SRAM-resident initialized array. RAM starts zeroed, `$8002-$bfff` is initialized to `$ff`, and `rom_image.h` is inserted at `$c000`:

```c
static uint8_t memory[64 * 1024] __attribute__((section(".data.memory_image"), aligned(4), used)) = {
    [0x8002 ... 0xbfff] = 0xff,
    [0xc000] =
#include "rom_image.h"
};
```

Replace `src/rom_image.h` with 16KB of comma-separated hex bytes. Offset `$0000` in that file maps to MPU address `$c000`. The included placeholder image is `JMP $C000` plus reset vector `$C000`.

## UART status bits

The `$8000` read status register only exposes the two MC6850-compatible flow bits needed by the bus loop:

| Bit | Meaning in this firmware |
| --- | --- |
| 0 | RDRF: USB-to-MPU SIO FIFO has at least one byte |
| 1 | TDRE: MPU-to-USB SIO FIFO is not full |

This maps directly to RP2040 `sio_hw->fifo_st & 0x03`, so the hot path reads the UART status with one SIO register load plus a two-bit mask. Writes to `$8000` are ignored. Interrupt output, DCD/CTS, and overrun status are not implemented.

## Timing model

Core 1 is dedicated to the MPU bus loop. It polls until E and VMA are both high, samples address/R/W via `sio_hw->gpio_in`, decodes only the two ACIA read addresses before falling through to `memory[address]`, and services the loop from SRAM-resident code. On MPU write cycles, the bus core waits for E to fall, then samples GPIO16-23 from that same E-low `gpio_in` snapshot before committing RAM or ACIA data writes.

The data GPIOs are assigned to PIO, not SIO. The data bus uses three PIO state machines: one continuously accepts low-byte values and writes GPIO16-23, one controls GPIO16-19 directions, and one controls GPIO20-23 directions. On each read cycle the bus core pushes only the data byte to the value SM, whose one-instruction `out pins, 8` loop uses autopull and stalls while the TX FIFO is empty. The two 4-bit direction SMs watch E, R/W, and VMA directly; when E is high and both R/W and VMA are high they drive their nibbles, then release GPIO16-23 to input/Hi-Z when E falls. This avoids bus contention on the next write cycle.

Core 0 owns USB CDC stdio and exchanges UART data with core 1 through the RP2040 SIO core-to-core FIFO. From the bus core's perspective, FIFO `VLD` is ACIA RDRF and FIFO `RDY` is ACIA TDRE. MPU reads from `$8001` never block. MC6850 software is expected to check RDRF first; if `$8001` is read while no byte is available, the returned value is intentionally undefined. MPU writes to `$8001` write one byte to the SIO FIFO when it has room; otherwise the byte is dropped.

GPIO29 clock output is generated by PIO instead of `clock_gpio_init()`, because RP2040 hardware clock outputs are limited to selected GPOUT pins. The firmware sets `clk_sys` to 128MHz, and the PIO clock program produces an 8MHz square wave with 8 system cycles per half-period.

## Build

Install and export the Pico SDK in the usual way, then build:

```sh
export PICO_SDK_PATH=/path/to/pico-sdk
cmake -S . -B build -DPICO_BOARD=pico
cmake --build build
```

The UF2 will be generated as:

```text
build/pico_mpu_6802.uf2
```

For timing validation, build with optimization enabled and check E, R/W, VMA, GPIO16-23, and GPIO29 on a logic analyzer. The firmware is structured for the requested half-clock service window, but the final margin depends on the exact board, wiring, level shifting, and MPU timing.
