# Pico + W65C816S bus firmware

Raspberry Pi Pico/RP2040 が外付け W65C816S に ROM、RAM、USB CDC UART を提供するための Pico SDK ファームウェアです。GPIO29 からクロックを供給し、GPIO24/26/27 は固定の非アクティブ制御信号として出力します。

## Pin map

| Pico GPIO | Direction | Signal |
| --- | --- | --- |
| 0-15 | in | Address bus A0-A15 |
| 16-23 | in/out, PIO controlled | Data bus D0-D7 |
| 24 | out | /NMI, fixed high |
| 25 | in | R/W |
| 26 | out | /IRQ, fixed high |
| 27 | out | RDY, fixed high |
| 28 | out | /RESET |
| 29 | out, PIO controlled | MPU clock |

Standard Pico boards do not bring every GPIO in this table out as a normal header pin. This pinout is most natural on a bare RP2040 or a board that exposes GPIO24 and GPIO29.

Tie any W65C816S control inputs that are not driven by this firmware, such as BE and /ABORT, to their inactive external levels as required by your board.

## Memory map

| MPU address | Function |
| --- | --- |
| `$0000-$1fff` | 8KB RAM, initialized to `$00` |
| `$3000` | 6850-like status register |
| `$3001` | 6850-like data register |
| `$6000-$7fff` | 8KB RAM, initialized to `$00` |
| `$8000-$ffff` | 32KB ROM image |
| other addresses | unmapped, reads return `$ff`, writes ignored |

The MPU-visible memory image is one 64KB SRAM-resident array. It is initialized to `$ff`, then the two independent RAM windows at `$0000-$1fff` and `$6000-$7fff` are initialized to `$00`. The ROM image is inserted into the upper half of the array:

```c
static uint8_t memory[64 * 1024] __attribute__((section(".data.memory_image"), aligned(4), used)) = {
    [0 ... MEMORY_SIZE - 1] = 0xff,
    [0 ... RAM_SIZE - 1] = 0x00,
    [MPU_RAM2_BASE ... MPU_RAM2_BASE + RAM_SIZE - 1] = 0x00,
    [0x8000] =
#include "rom_image.h"
};
```

`src/rom_image.h` currently includes `../../ROM/ROM6502.h`. Replace it with up to 32KB of comma-separated hex bytes if you want a private 65816 Pico ROM image. Offset `$0000` in that file maps to MPU address `$8000`.

## UART status bits

The `$3000` read status register only exposes the two MC6850-compatible flow bits needed by the bus loop:

| Bit | Meaning in this firmware |
| --- | --- |
| 0 | RDRF: USB-to-MPU SIO FIFO has at least one byte |
| 1 | TDRE: MPU-to-USB SIO FIFO is not full |

This maps directly to RP2040 `sio_hw->fifo_st & 0x03`, so the hot path reads the UART status with one SIO register load plus a two-bit mask. Writes to `$3000` are ignored. Interrupt output, DCD/CTS, and overrun status are not implemented.

## Timing model

Core 1 is dedicated to the MPU bus loop. It polls GPIO29, which is also the PIO-generated MPU clock output, and samples address/R/W while the clock is high. On read cycles it decodes `$3000/$3001` before reading directly from the memory image. On write cycles it waits for the clock to fall, samples GPIO16-23, and commits RAM or ACIA data writes.

The data GPIOs are assigned to PIO, not SIO. The data bus uses three PIO state machines: one continuously accepts low-byte values and writes GPIO16-23, one controls GPIO16-19 directions, and one controls GPIO20-23 directions. The two 4-bit direction SMs watch GPIO29 and R/W directly; when the clock is high and R/W is high they drive their nibbles, then release GPIO16-23 to input/Hi-Z when the clock falls.

Core 0 owns USB CDC stdio and exchanges UART data with core 1 through the RP2040 SIO core-to-core FIFO. From the bus core perspective, FIFO `VLD` is ACIA RDRF and FIFO `RDY` is ACIA TDRE. MPU reads from `$3001` never block. MC6850 software is expected to check RDRF first; if `$3001` is read while no byte is available, the returned value is intentionally undefined. MPU writes to `$3001` write one byte to the SIO FIFO when it has room; otherwise the byte is dropped.

GPIO29 clock output is generated by PIO instead of `clock_gpio_init()`, because RP2040 hardware clock outputs are limited to selected GPOUT pins. The firmware raises VREG to 1.25V, sets `clk_sys` to 295.2MHz, and the PIO clock program uses `set pins, ... [25]` with clkdiv `3 + 44/256`, producing `295.2MHz / (((1 + 25) + (1 + 25)) * (3 + 44/256))`, about 1.789MHz.

## Build

Install and export the Pico SDK in the usual way, then build:

```sh
export PICO_SDK_PATH=/path/to/pico-sdk
cmake -S . -B build -DPICO_BOARD=pico
cmake --build build
```

The UF2 will be generated as:

```text
build/pico_mpu_65816.uf2
```

For timing validation, build with optimization enabled and check GPIO29 clock, R/W, GPIO16-23, /RESET, /NMI, /IRQ, and RDY on a logic analyzer. The final margin depends on the exact board, wiring, level shifting, and W65C816S timing.
