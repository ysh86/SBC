# Pico + HD63C09E bus firmware

Raspberry Pi Pico/RP2040 が外付け HD63C09E に ROM、RAM、USB CDC UART を提供するための Pico SDK ファームウェアです。

## Pin map

| Pico GPIO | Direction | Signal |
| --- | --- | --- |
| 0-15 | in | A0-A15 |
| 16-23 | in/out, PIO controlled | D0-D7 |
| 24 | out | /NMI, held high |
| 25 | in | R/W |
| 26 | out | /IRQ, held high |
| 27 | out, PIO controlled | Q, 3MHz MPU clock |
| 28 | out | /RESET |
| 29 | out, PIO controlled | E, 3MHz MPU clock |

Standard Pico boards do not bring every GPIO in this table out as a normal header pin. This pinout is most natural on a bare RP2040 or a board that exposes GPIO24 and GPIO29.

## Memory map

現状は 6802 版から維持しています。

| MPU address | Function |
| --- | --- |
| `$0000-$dfff` | 56KB RAM, zero-filled |
| `$e000` | 6850-like status register, writes ignored |
| `$e001` | 6850-like data register |
| `$e002-$efff` | I/O window, reads return `$ff`, writes ignored |
| `$f000-$ffff` | 4KB ROM |

The MPU-visible memory image is one 64KB SRAM-resident initialized array. RAM starts zeroed, the `$e000-$efff` I/O window is initialized to `$ff`, and `rom_image.h` is inserted at `$f000`:

```c
static uint8_t memory[64 * 1024] __attribute__((section(".data.memory_image"), aligned(4), used)) = {
    [0xe000 ... 0xefff] = 0xff,
    [0xf000] =
#include "rom_image.h"
};
```

Replace `src/rom_image.h` with 4KB of comma-separated hex bytes. Offset `$0000` in that file maps to MPU address `$f000`. The included placeholder image is `JMP $F000` plus reset vector `$F000`.

## UART status bits

The `$e000` read status register only exposes the two MC6850-compatible flow bits needed by the bus loop:

| Bit | Meaning in this firmware |
| --- | --- |
| 0 | RDRF: USB-to-MPU SIO FIFO has at least one byte |
| 1 | TDRE: MPU-to-USB SIO FIFO is not full |

This maps directly to RP2040 `sio_hw->fifo_st & 0x03`, so the hot path reads the UART status with one SIO register load plus a two-bit mask. Writes to `$e000` are ignored. Interrupt output, DCD/CTS, and overrun status are not implemented.

## Timing model

Core 1 is dedicated to the MPU bus loop. It polls until E is high, samples address/R/W via `sio_hw->gpio_in`, decodes only the two ACIA read addresses before falling through to `memory[address]`, and services the loop from SRAM-resident code. On MPU write cycles, the bus core waits for E to fall, then samples GPIO16-23 from that same E-low `gpio_in` snapshot before committing RAM or ACIA data writes.

The data GPIOs are assigned to PIO, not SIO. The data bus uses three PIO state machines: one continuously accepts low-byte values and writes GPIO16-23, one controls GPIO16-19 directions, and one controls GPIO20-23 directions. On each read cycle the bus core pushes only the data byte to the value SM, whose one-instruction `out pins, 8` loop uses autopull and stalls while the TX FIFO is empty. The two 4-bit direction SMs watch E and R/W directly; when E is high and R/W is high they drive their nibbles, then release GPIO16-23 to input/Hi-Z when E falls. This avoids bus contention on the next write cycle.

GPIO27/GPIO29 Q/E clock output is generated by PIO. The firmware raises the RP2040 core voltage to 1.15V, sets `clk_sys` to 240MHz, and the PIO clock program produces 3MHz quadrature Q/E clocks with 20 system cycles per phase.

## Build

Install and export the Pico SDK in the usual way, then build:

```sh
export PICO_SDK_PATH=/path/to/pico-sdk
cmake -S . -B build -DPICO_BOARD=pico
cmake --build build
```

The UF2 will be generated as:

```text
build/pico_mpu_6309E.uf2
```

For timing validation, build with optimization enabled and check E, Q, R/W, and GPIO16-23 on a logic analyzer. The firmware is structured for the requested clock rate, but the final margin depends on the exact board, wiring, level shifting, and MPU timing.
